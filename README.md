# LinkMus Monitor

Система мониторинга серверов и рабочих станций для гетерогенных корпоративных сетей. Агенты собирают метрики и отправляют их на центральный сервер, веб-интерфейс отображает состояние всей инфраструктуры в реальном времени.

---

## Возможности

- **Поддержка Linux и Windows** — агент компилируется под amd64/arm64 (Linux) и amd64 (Windows)
- **Дашборд** — карточки всех узлов с CPU, RAM, диском, сетью и статусом сервисов; drag-and-drop сортировка, псевдонимы узлов
- **Детальная страница узла** — графики, процессы, диски, сетевые интерфейсы, TCP, SNMP, FSRM-квоты
- **История** — 1ч / 7д / 14д / 30д с адаптивным усреднением
- **Пробы сервисов** — SSH, RDP, SMB, HTTP, WinRM + произвольные TCP-сервисы с временем отклика
- **Алерты** — CPU, RAM, узел offline/online через Email и Telegram
- **AI-отчёты** — анализ инфраструктуры через GigaChat с историей и экспортом в .txt
- **Экспорт метрик** — CSV или JSON из браузера
- **Авторизация** — вход по логину/паролю, сессионные токены

## Архитектура

```
  Агенты (mon-agent)          Мастер-сервер (mon-server)
  ┌─────────────────┐         ┌──────────────────────────┐
  │ Linux / Windows │──POST──▶│ Go + SQLite (WAL)         │
  │ сбор метрик     │         │ /api/metrics  (агенты)    │
  │ каждые N сек    │         │ /api/nodes    (фронтенд)  │
  └─────────────────┘         │ /api/auth/*               │
                              └────────────┬─────────────┘
                                           │
                              ┌────────────▼─────────────┐
                              │ React + Vite + Recharts   │
                              │ Tailwind CSS, тёмная тема │
                              └──────────────────────────┘
```

**Push-модель:** агент сам инициирует соединение → сервер не требует доступа к агентам, работает через NAT.

## Установка сервера (Linux, amd64)

```bash
curl -sSL https://raw.githubusercontent.com/RomanMusalikin/linkmus-monitor/main/install.sh | sudo bash
# Выбрать пункт 1 — Установить / обновить сервер
```

Скрипт скачает бинарник и фронтенд из последнего GitHub Release, создаст systemd-службу `mon-server` и настроит обратный прокси (Nginx или Caddy) при наличии. При повторном запуске — обновит до новой версии.

После установки откройте `http://<IP-сервера>` — при первом входе система предложит создать учётную запись администратора.

### Тонкости установки

**Порт занят.** По умолчанию используется порт `8080`. Если он занят (например, другим сервисом), установщик спросит альтернативный порт. Перед установкой проверьте: `ss -tlnp | grep 8080`.

**Caddy в Docker.** Если Caddy запущен как контейнер (без локального бинарника), установщик автоматически находит контейнер через `docker ps`, читает путь к Caddyfile из `docker inspect` и выполняет `docker exec caddy caddy reload`. Убедитесь, что контейнер называется `caddy`.

**UFW + Docker.** При добавлении правила UFW для Docker-Caddy скрипт автоматически перезапускает Docker, чтобы восстановить его iptables-цепочки. Если UFW не активен — Docker не трогается.

**Агент на том же сервере.** Если агент устанавливается на ту же машину, что и мастер-сервер, указывайте в конфиге агента не публичный домен, а `http://127.0.0.1:<порт>`. Это надёжнее и не зависит от DNS. Конфиг: `/opt/mon-agent/agent-config.yaml`.

## Управление сервером

```bash
mon server start|stop|restart   # запуск / остановка / перезапуск
mon server status                # статус службы
mon server logs                  # логи в реальном времени (journalctl -f)
mon server update                # обновить до последней версии
mon server rollback              # откатить к предыдущей версии
mon server delete                # полностью удалить (служба, файлы, база данных)
```

## Установка агента

### Linux (amd64 / arm64)

```bash
curl -sSL https://raw.githubusercontent.com/RomanMusalikin/linkmus-monitor/main/install.sh | sudo bash
# Выбрать пункт 2 — Установить / обновить агент Linux
```

Скрипт спросит URL сервера (`http://IP:порт` или `https://домен`), создаст systemd-службу `mon-agent`. Путь `/api/metrics` добавляется автоматически — вводить не нужно.

**Агент на той же машине, что и сервер:** укажите `http://127.0.0.1:<порт>`.

Управление:

```bash
mon agent start|stop|restart
mon agent status
mon agent logs      # все логи через journalctl, включая ошибки соединения
mon agent update
mon agent rollback
mon agent delete
```

### Windows (amd64)

1. Скачайте `mon-agent-windows-amd64.zip` со страницы релизов
2. Распакуйте — внутри `mon-agent.exe` и `install-agent.ps1`
3. Запустите PowerShell от Администратора в папке с файлами:

```powershell
Set-ExecutionPolicy Bypass -Scope Process -Force; .\install-agent.ps1
```

Скрипт создаст конфиг `C:\mon-agent\agent-config.yaml` и зарегистрирует службу `MonAgent` через `New-Service` с автозапуском.

Логи агента: `C:\mon-agent\mon-agent.log` (ротация при достижении 5 МБ).

```powershell
mon agent start|stop|restart|status|logs|update
```

Если команда `mon` не найдена сразу после установки — откройте новый терминал.

## Метрики агента

| Категория | Метрики |
|-----------|---------|
| CPU | загрузка %, user/system/iowait/steal %, температура (Linux), по ядрам, модель, частота, load avg (Linux) |
| RAM | используется/всего ГБ, кэш, буферы, swap |
| Диски | % по корневому разделу, все разделы (точка монтирования, размер, использование), I/O: чтение/запись байт/с, очередь |
| Сеть | входящий/исходящий трафик байт/с по основному интерфейсу и всем интерфейсам |
| TCP | всего соединений, Established, TIME_WAIT |
| Процессы | количество, top-10 по CPU, top-10 по RAM (PID, имя, %, МБ, пользователь) |
| Система | hostname, OS, IP, uptime, дата загрузки, кол-во залогиненных пользователей |
| Windows | статус RDP/SMB, FSRM-квоты (путь, лимит, использование, нарушения) |

## Пробы сервисов (server-side)

Сервер опрашивает каждый узел по его IP каждые **15 секунд**. Агент для этого не нужен — все пробы выполняются на стороне сервера.

### Встроенные сервисы

| Сервис | Порт по умолчанию | Примечание |
|--------|-------------------|------------|
| SSH | 22 | Все узлы |
| HTTP | 80 | Все узлы |
| RDP | 3389 | Только Windows-узлы |
| SMB | 445 | Только Windows-узлы |
| WinRM | 5985 | Только Windows-узлы |

Метод — TCP connect с таймаутом 3 секунды. HTTP дополнительно делает GET-запрос для получения кода ответа.

### Настраиваемые порты

В **Настройках** задаются глобальные порты для всех узлов. На странице конкретного узла можно переопределить порт индивидуально. Пробер использует per-node порт, если он задан.

### Пользовательские TCP-сервисы

В Настройках → «Порты сервисов» добавляются произвольные сервисы (имя + порт). Они проверяются на всех узлах. Для каждого узла можно:
- скрыть ненужные сервисы (редактор видимости)
- задать другой порт (per-node переопределение)

### Настройка видимости

На странице узла в разделе «Сервисы» — кнопка «Настроить отображение». Для Linux-узлов Windows-специфичные сервисы (RDP, SMB, WinRM) из списка скрыты автоматически.

## SNMP

Опрос по UDP, порт 161, SNMPv2c, community `public`, каждые **30 секунд**.

| OID | Данные |
|-----|--------|
| sysUpTime | Время работы устройства |
| sysName | Имя хоста по SNMP |
| hrProcessorLoad | Загрузка CPU |
| ifNumber | Количество интерфейсов |
| ifTable | Имя, скорость линка, входящий/исходящий трафик байт/с |

Если устройство не отвечает на SNMP — раздел на странице узла не отображается.

## Алерты

Настраиваются в Настройках → «Алерты». Поддерживаются Email (SMTP) и Telegram.

| Условие | Когда срабатывает |
|---------|-------------------|
| CPU | Загрузка ≥ порога (%) |
| RAM | Использование ≥ порога (%) |
| Узел offline | Узел перестал отвечать (нет метрик > 30 сек) |
| Узел online | Узел восстановился после offline |

**Кулдаун** (по умолчанию 30 мин) применяется к CPU и RAM алертам — предотвращает спам при длительной высокой нагрузке. Алерты смены статуса (offline/online) срабатывают на каждый реальный переход.

### Telegram

Укажите токен бота и Chat ID. Опционально — Topic ID для групп с топиками (форум-режим).

### Email

Поддерживаются порты 465 (implicit TLS) и 587 (STARTTLS).

## AI-отчёты (GigaChat)

Настраиваются в Настройках → «GigaChat» (Client ID + Client Secret от Sberbank GigaChat API).

Отчёт генерируется по выбранному периоду: 1ч / 6ч / 12ч / 24ч / 7д / 30д или произвольному диапазону дат. Содержит:
- общий статус инфраструктуры
- анализ по каждому узлу (CPU, RAM, диск, сеть)
- выявленные проблемы
- рекомендации

История последних 100 отчётов сохраняется в базе, каждый можно открыть и экспортировать в `.txt`.

## История и графики

| Диапазон | Детализация |
|----------|-------------|
| 1 час | 10-минутное усреднение |
| 7 дней | 30-минутное усреднение |
| 14 дней | 30-минутное усреднение |
| 30 дней | Часовое усреднение |

Хранится: сырые метрики ~25 часов, 30-минутные и часовые агрегаты — 90 дней.

Графики: CPU, RAM, диск, сеть (входящий/исходящий), дисковый I/O.

## Разработка

```bash
# Сервер (Go)
go run ./cmd/server/

# Фронтенд (Vite проксирует /api/* → localhost:8080)
cd web && npm install && npm run dev

# Кросс-компиляция агентов
CGO_ENABLED=0 GOOS=linux   GOARCH=amd64 go build -o dist/mon-agent-linux        ./cmd/agent/
CGO_ENABLED=0 GOOS=linux   GOARCH=arm64 go build -o dist/mon-agent-linux-arm64  ./cmd/agent/
CGO_ENABLED=0 GOOS=windows GOARCH=amd64 go build -o dist/mon-agent.exe          ./cmd/agent/

# Сервер
CGO_ENABLED=0 GOOS=linux GOARCH=amd64 go build -o dist/mon-server ./cmd/server/
cd web && npm run build  # → web/dist/
```

Переменные окружения сервера:

| Переменная | По умолчанию | Назначение |
|------------|--------------|------------|
| `PORT` | `8080` | Порт прослушивания |
| `DB_PATH` | `monitor.db` | Путь к SQLite базе |
| `WEB_PATH` | `/opt/linkmus-monitor/web` | Путь к статике фронтенда |
